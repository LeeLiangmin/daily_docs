# 现状-目标-路径-计划（Situation → Goal → Approach → Plan）

一个通用的任务梳理框架。核心目的：在动手做之前，先想清楚“现在在哪、要去哪、怎么去、具体怎么走”，避免过早陷入执行细节或想到哪做到哪。

## 什么时候用 / 什么时候不用

**用**：目标模糊、涉及多个步骤或多方利益相关者、存在不确定性或多种可行路径、用户明确要求梳理/规划的任务。

**不用**：单一、明确、一步就能完成的任务（比如"把这段代码里的 bug 修一下"）。这种情况直接做即可，套用框架反而拖慢节奏。如果拿不准，可以先问自己：这个任务如果不拆解，Claude 会不会漏掉重要的权衡或步骤？会，就用；不会，就直接做。

## 四个阶段

### 1. 现状梳理（Situation）

目的是建立一个**双方都认可的事实基线**，而不是急着下结论或给方案。

- 先看对话历史和已有材料（上传的文件、之前提到的背景）里已经包含的信息，能提取的先提取，不要重复问用户已经说过的内容。
- 缺口部分，用简短、聚焦的问题补齐，一次别问太多，优先问会实质影响后续路径选择的问题（而不是无关紧要的细节）。
- 现状梳理至少覆盖：
  - **现在的实际情况**：已经有什么、进展到哪一步了
  - **可用资源和约束**：时间、预算、人力、技术条件、外部依赖
  - **已知的痛点/障碍**：是什么让现状还没达到目标
- 只陈述可验证或用户已确认的事实，不要把猜测当成现状写进去；不确定的地方明确标注"待确认"。

### 2. 目标定义（Goal）

- 把目标从模糊描述（"我想变得更好"、"提升效率"）转成**可判断是否达成**的描述。可以借助 SMART（具体、可衡量、可达成、相关、有时限）作为检查清单，但不必生搬硬套格式。
- 区分**终极目标**（为什么做这件事）和**这一轮要交付的目标**（这次具体要做到什么程度），两者不一样时要分别写出来。
- 明确"怎么算成功"——如果无法用一两句话说清楚验收标准，说明目标还不够清晰，需要回到用户那里进一步澄清，而不是自己脑补一个。

### 3. 头脑风暴与路径讨论（Brainstorm & Approach）

这一步是现状梳理和制定计划之间承上启下的部分，也是最容易被跳过、却最容易埋雷的地方——很多时候问题不是"计划写得不够细"，而是"路径压根没和用户讨论过就定了"。这一步分三个动作，顺序不要打乱：

**a. 发散（Brainstorm）**
- 先尽量多地想可能的思路，不急于评判优劣，哪怕看起来不成熟、代价大、甚至互相矛盾的方案也先列出来，不要在发散阶段自我审查。
- 明确告诉用户这是"讨论草稿"而不是"最终方案"，降低用户否定或推翻的心理门槛，鼓励用户补充你没想到的选项。
- 如果任务复杂、涉及专业判断，可以主动指出发散时用到的角度（比如从成本、从时间、从风险偏好、从团队能力等不同维度各想一版）。

**b. 讨论（Discuss）**
- 把发散出来的选项摆出来给用户看，而不是自己在心里筛完只呈现"最优解"。
- 针对有分歧或不确定的地方，直接问用户的偏好和取舍（比如"更看重速度还是稳妥"），不要替用户假设答案。
- 这一步的目标是达成**双方都认可的方向**，不是产出一份漂亮的方案对比文档。

**c. 收敛（Converge）**
- 讨论清楚后，收敛成 2-3 条真正可行的路径，简要说明每条的核心思路、优势、代价或风险。
- 标出其中的**关键决策点**——哪些选择一旦定了会影响后面所有步骤（技术选型、组织架构调整、要不要分阶段上线等）。
- 如果某条路径依赖用户还没给的信息（预算上限、风险偏好等），标注清楚，不要假设一个答案往下走。

**检查点**：在路径没有和用户讨论、确认之前，不要直接跳到"制定计划"。如果用户还在犹豫或者明确说"再想想"，就留在这一步，不要为了推进而强行收尾。

### 4. 制定计划（Plan）

在前三步基础上，产出真正可执行的计划：

- **分阶段/分任务**：每个阶段有清晰的产出物和一个大致的工作量估计（不追求精确到小时，但要有相对量级：小/中/大，或者天/周）。
- **里程碑和验收标准**：每个阶段做完之后，怎么知道它做对了。
- **风险和应对**：列出最可能让计划出问题的 2-3 个风险点，以及对应的缓解思路，不用做成大而全的风险登记表。
- **下一步行动**：明确"看完这份计划后，接下来第一件事是什么"，避免计划做完了但没人知道从哪开始。

## 输出格式

默认按下面的结构组织回复（对话中直接给，或者根据 `docx`/`md` 技能存成文件，取决于用户需求和任务的分量）。注意：下面的"可行路径"是**讨论收敛之后**的结果；发散和讨论的过程本身是对话中来回进行的，不需要、也不应该把中间那些不成熟的选项都写进最终输出——最终文档只呈现收敛后的 2-3 条路径和关键决策点。

```
## 现状
- ...

## 目标
- 终极目标 / 本轮目标
- 成功标准

## 可行路径
### 路径 A：...
- 思路 / 优势 / 代价
### 路径 B：...
...
（如需用户决策的关键点，单独标出）

## 计划
### 阶段 1：...
- 产出物 / 工作量 / 验收标准
### 阶段 2：...
...

## 风险与应对
- ...

## 下一步
- ...
```

篇幅要跟任务的复杂度匹配——简单任务几句话带过每个板块即可，不要为了凑结构而注水。如果是要交付的正式文档，参考 `references/template.md` 里更完整的模板结构。

## 使用中的原则

- **先现状，后方案**：如果发现自己还没搞清楚现状就已经在心里定了计划，退回去先把现状和目标确认清楚。
- **先讨论，后计划**：不要在头脑风暴环节自问自答就直接收敛出计划，路径需要真正和用户过一遍，尤其是有分歧或者依赖用户偏好的地方。
- **路径要摆出来，不要替用户悄悄选**：尤其是关键决策点，宁可多问一句，也不要自己直接拍板走了一条路径。
- **计划要能落地**：避免抽象的"加强""优化""推进"这类无法执行验收的措辞，每个阶段尽量落到"谁/做什么/怎么算完成"。
- **保持轻量**：这是一个思考框架，不是要把每次对话都变成一份沉重的报告。任务越简单，走这四步的痕迹应该越淡。
