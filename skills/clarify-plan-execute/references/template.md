# 正式版模板（用于需要交付文档的场景）

当任务足够复杂、正式，需要产出一份可以发给他人评审的文档时，使用下面更完整的结构；否则用 SKILL.md 里的轻量版即可。

```markdown
# [任务/项目名称]

**日期**：YYYY-MM-DD
**状态**：梳理中 / 待评审 / 已确认

## 一、背景与现状（已核实 / 待确认分开标注）

### 1.1 现在的情况
- 已经完成 / 存在的部分
- 目前所处的阶段

### 1.2 资源与约束
- 时间
- 预算 / 人力
- 技术或组织限制

### 1.3 主要痛点
- 痛点 1：具体描述 + 影响
- 痛点 2：...

## 二、目标

### 2.1 终极目标
一句话说明为什么做这件事、解决什么问题。

### 2.2 本轮目标
这次具体要交付/达成到什么程度。

### 2.3 成功标准
- 标准 1（可验证）
- 标准 2（可验证）

## 三、可行路径

### 路径 A：[名称]
- 核心思路
- 优势
- 代价 / 风险
- 适用条件

### 路径 B：[名称]
（同上结构）

### 关键决策点
- 决策点 1：需要谁来定、什么时候必须定下来
- 决策点 2：...

### 推荐路径
基于当前信息的推荐，以及做出这个推荐的理由（不代替用户做最终决定，只是给出建议）。

## 四、难题与风险评估

### 4.1 已知难题 / 权衡取舍
这类是路径本身自带的、确定存在的约束，不是"可能发生"而是"选了就要面对"，没有预防措施可言。

| 取舍点 | 选择 | 理由 | 放弃一边的代价 |
|---|---|---|---|
| 例：性能 vs 成本 | ... | ... | ... |

### 4.2 不确定性风险
针对上面选定/推荐的路径展开，覆盖执行、资源、外部、决策四类风险，只列真正值得关注的条目。

| 风险 | 类别 | 影响 | 概率 | 预防措施 | 应对/补救方案 | 触发信号 |
|---|---|---|---|---|---|---|
| ... | 执行/资源/外部/决策 | 高/中/低 | 高/中/低 | ... | ... | ... |

## 五、实施计划

### 阶段 1：[名称]（预计工作量：S/M/L）
- 任务列表
- 产出物
- 验收标准
- 依赖
- 相关取舍/风险检查点（对应上面哪几条难题或风险）

### 阶段 2：[名称]
（同上结构，按需增加阶段）

## 六、下一步行动
- 立即要做的第一件事
- 谁负责 / 什么时候看下一次进展
```

## 使用建议

- 表格和分级（S/M/L、高/中/低）是为了让评审者快速抓重点，不是为了看起来"专业"——如果内容量不足以支撑表格，就用一两句话代替。
- 这份模板偏组织/项目场景；如果是个人任务（比如学习计划、找工作规划），可以把"资源与约束""难题与风险评估"简化甚至省略，保留"现状-目标-路径-计划"这条主线即可。
- 这份文档对应的是"计划"阶段的产出。计划确认完就停下来，是否进入执行由使用者决定；一旦同意执行，进度应记录在 `.progress/` 目录下，而不是靠这份静态文档去追踪动态状态。
