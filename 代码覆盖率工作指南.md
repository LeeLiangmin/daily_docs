# 代码覆盖率工具使用指南（lcov / Clang）

本文整理了使用 lcov 收集 C/C++ 覆盖率、开启分支覆盖率、判断 lcov 版本，以及用 Clang 做覆盖率插桩的完整流程。

---

## 一、lcov 基本流程回顾

lcov 是 GCC 自带覆盖率工具 **gcov 的前端**：gcov 生成原始数据，lcov 把数据汇总成 `.info` 文件，再用 `genhtml` 渲染成网页报告。典型四步：

```bash
# 1. 编译时插桩（关掉优化，避免行号错位）
gcc --coverage -O0 -g -o myprog myprog.c

# 2. 运行程序 / 测试，产出 .gcda 数据
./myprog

# 3. 采集，汇总成 .info（顺便剔除系统头文件、测试代码）
lcov --capture --directory . --output-file coverage.info
lcov --remove coverage.info '/usr/*' '*/test/*' --output-file coverage.info

# 4. 生成 HTML 报告
genhtml coverage.info --output-directory out
```

浏览器打开 `out/index.html` 即可查看分层的覆盖率，红色为未覆盖行。

---

## 二、为什么没有分支覆盖率

**核心原因：lcov 默认不采集、也不显示分支覆盖率，需要手动开启，而且要在「采集」和「生成 HTML」两个阶段都开。**

`branch_coverage` 选项默认值为 `0`（关闭它能减少内存/CPU 开销并缩小数据文件）。genhtml 同理：默认报告只含行覆盖和函数覆盖，分支（及 MC/DC）数据默认不显示。

### 关键陷阱

很多人只在 `genhtml` 阶段加了开关，结果仍然没有分支数据。原因是分支信息在 `lcov --capture` 阶段就被丢掉了——采集时没开，生成的 `.info` 里根本没有分支记录（`BRDA` / `BRF` / `BRH` 那几行），genhtml 再怎么加参数也变不出来。**所以采集和渲染都要开。**

### 新版 lcov（2.x，推荐写法）

```bash
lcov --capture --directory . --branch-coverage --output-file coverage.info
genhtml coverage.info --branch-coverage --output-directory out
```

`--branch-coverage` 等价于 `--rc branch_coverage=1`。

### 旧版 lcov（1.x）

旧版用的是带前缀的 rc 名字：

```bash
lcov --capture --directory . --rc lcov_branch_coverage=1 --output-file coverage.info
genhtml coverage.info --rc genhtml_branch_coverage=1 --output-directory out
```

`lcov_branch_coverage` / `genhtml_branch_coverage` 在新版仍兼容，但已被标记 deprecated。

### 一劳永逸：写进 lcovrc

把它写进用户级配置 `~/.lcovrc`（命令行选项会覆盖配置文件）：

```
branch_coverage = 1
```

之后普通的 `lcov --capture` 和 `genhtml` 就自带分支覆盖。

### 开了之后分支数据仍然不对

- **编译别开优化**：`-O2` / `-O3` 会折叠、合并分支，导致行号和分支对不上或数量异常。覆盖率构建统一用 `-O0 -g --coverage`。
- **C++ 的「幽灵分支」**：异常处理会让每行凭空多出一堆分支，分母被撑大、覆盖率虚低。可用 lcov 的 bogus branch filtering（如 `--filter branch`）过滤伪分支。
- **Clang 工具链**：gcov 版本要和编译器匹配，否则采集不到完整分支数据，需走 `llvm-cov gcov` 或 LLVM 自己的 `llvm-cov`（见第四节）。

## 三、如何判断 lcov 是新版还是旧版

最直接的办法：

```bash
lcov --version
```

输出形如 `lcov: LCOV version 2.0-1` 或 `lcov: LCOV version 1.16`。**2.x 及以上为「新版」，1.x 为「旧版」，分界点是 2.0。**

其他间接判断方式：

- **用 help 看选项**：`lcov --help | grep -i branch`。能搜到裸选项 `--branch-coverage` / `--no-branch-coverage` 基本就是 2.x；只能用 `--rc lcov_branch_coverage=1` 这种 rc 形式则多半是 1.x。
- **看 rc 选项名**：旧版用带前缀的 `lcov_branch_coverage` / `genhtml_branch_coverage`，新版统一为不带前缀的 `branch_coverage`。
- **包管理器查询**：

  ```bash
  dpkg -l lcov        # Debian/Ubuntu
  brew info lcov      # macOS
  rpm -q lcov         # 红帽系
  ```

经验判断：lcov 2.0 于 2022 年发布，近两三年装的系统基本是 2.x，老服务器或长期支持版更可能仍是 1.1x——但**以 `lcov --version` 实际输出为准**。

> 注意：两个版本差异不只分支覆盖一处，2.x 在差异覆盖（differential coverage）、并行处理、MC/DC 等方面都有较大改动，命令行参数也调整较多。网上抄来的命令跑不通，很多是新旧混用导致的「参数不识别」。

## 四、Clang 覆盖率插桩

Clang 有**两套**方案，选哪套取决于你想要 LLVM 原生（更精确）的报告，还是沿用 lcov 工具链。

### 方案一：Source-based coverage（LLVM 原生，推荐）

最精确，能给到行、region（区域）、分支，新版还支持 MC/DC。流程为「插桩编译 → 运行产出 profraw → profdata 索引 → llvm-cov 出报告」。

**1. 编译时加两个 flag：**

```bash
clang -fprofile-instr-generate -fcoverage-mapping -O0 -g foo.c -o foo
```

`-fprofile-instr-generate` 插入计数器，`-fcoverage-mapping` 生成源码到计数器的映射（缺了它只有 PGO 数据、无法出覆盖率报告）。C++ 换 `clang++`，每个编译单元和链接阶段都要带这两个 flag。

**2. 运行程序，产出原始数据：**

```bash
LLVM_PROFILE_FILE="foo.profraw" ./foo
```

多进程/多次运行时用占位符避免互相覆盖：`%p` 是进程 PID，`%m` 按签名自动合并到同一池子。例如：

```bash
export LLVM_PROFILE_FILE="prof/%p.profraw"
```

**3. 合并、索引成 profdata：**

```bash
llvm-profdata merge -sparse foo.profraw -o foo.profdata
# 多文件：llvm-profdata merge -sparse prof/*.profraw -o foo.profdata
```

`-sparse` 丢掉零计数项、缩小文件，单纯出报告很合适。

**4. 用 llvm-cov 出报告（三种形态）：**

```bash
llvm-cov report ./foo -instr-profile=foo.profdata          # 终端汇总表
llvm-cov show   ./foo -instr-profile=foo.profdata \
    -format=html -output-dir=out                           # 逐行 HTML
llvm-cov export ./foo -instr-profile=foo.profdata \
    -format=lcov > coverage.info                           # 导出成 lcov .info
```

注意 `llvm-cov` 的第一个参数是**可执行文件本身**（从二进制读取 coverage mapping），`-instr-profile` 才是数据。

- 看分支：`llvm-cov show ... -show-branches=count`
- 看 MC/DC：编译时加 `-fcoverage-mcdc`（Clang 18+），show 时加 `-show-mcdc`

### 方案二：gcov 兼容插桩（要接 lcov 时用）

沿用 lcov + genhtml 流程，走 gcov 兼容模式，和 GCC 的 `--coverage` 一致：

```bash
clang --coverage -O0 -g foo.c -o foo      # 等价 -fprofile-arcs -ftest-coverage
./foo                                      # 产出 .gcno / .gcda
```

**关键：读取时用 `llvm-cov gcov` 而不是系统 gcov。** lcov 通过 `--gcov-tool` 指过去：

```bash
lcov --capture --directory . --gcov-tool llvm-cov-gcov \
     --branch-coverage --output-file coverage.info
genhtml coverage.info --branch-coverage --output-directory out
```

由于 `llvm-cov gcov` 是**子命令**形式，而 lcov 期望一个单独可执行，常见做法是包一层小脚本：

```bash
# 存成 llvm-cov-gcov，chmod +x
#!/bin/sh
exec llvm-cov gcov "$@"
```

然后 `--gcov-tool ./llvm-cov-gcov`。

### 怎么选

- **新项目 / 想要更准的分支、MC/DC 数据** → 方案一，再用 `llvm-cov export -format=lcov` 喂给 genhtml 或上传 Codecov，既精确又复用 lcov 生态。
- **已有大量 lcov/gcov 流水线、不想改** → 方案二。

### 两个提醒

- 插桩构建一律 `-O0`，优化会让计数和源码行对不上。
- `llvm-profdata`、`llvm-cov` 的版本最好和 `clang` 来自同一套 LLVM 安装，跨大版本混用偶尔会报 profile 格式不兼容。

## 附：快速对照表

| 场景 | 编译 flag | 数据文件 | 出报告工具 |
|------|-----------|----------|------------|
| GCC + lcov | `--coverage -O0 -g` | `.gcno` / `.gcda` | `lcov` + `genhtml` |
| Clang 原生覆盖率 | `-fprofile-instr-generate -fcoverage-mapping -O0 -g` | `.profraw` → `.profdata` | `llvm-cov` |
| Clang 接 lcov | `--coverage -O0 -g` | `.gcno` / `.gcda` | `lcov --gcov-tool llvm-cov-gcov` + `genhtml` |

**分支覆盖率记得在采集和渲染两端都加 `--branch-coverage`。**
